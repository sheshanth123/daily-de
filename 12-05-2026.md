

## 1. Python & Data Manipulation

### Converting a Dictionary to a DataFrame

In Pandas, you typically use the `pd.DataFrame()` constructor or `from_dict()`. The structure of your dictionary (list of dicts vs. dict of lists) determines the approach.

```python
import pandas as pd

data = {
    'employee_id': [1, 2, 3],
    'dept': ['Data', 'Sales', 'Data'],
    'salary': [100000, 80000, 110000]
}

# Standard constructor
df = pd.DataFrame(data)

# If data is a list of records
records = [{'id': 1, 'name': 'A'}, {'id': 2, 'name': 'B'}]
df_records = pd.DataFrame.from_records(records)

```

### Slicing and Dicing (Pandas/NumPy)

The key here is understanding `.loc` (label-based) vs `.iloc` (integer-based).

* **Slicing rows and columns:** `df.loc[0:5, ['column_a', 'column_b']]`
* **Boolean Indexing:** `df[df['salary'] > 90000]`
* **NumPy Slicing:** `array[start:stop:step]`

### Simple Aggregation

To find the count of employees per department:

```python
# Pandas approach
dept_counts = df.groupby('dept')['employee_id'].count().reset_index()

# Alternative using value_counts
dept_counts = df['dept'].value_counts()

```

## 2. SQL & Analytics

### Monthly Spend & Growth Percentage

This is a classic "Window Function" test. You need to calculate the current month and compare it to the previous one using `LAG()`.

```sql
WITH monthly_sales AS (
    SELECT 
        customer_id,
        DATE_TRUNC('month', transaction_date) AS sales_month,
        SUM(amount) AS current_month_spend
    FROM transactions
    GROUP BY 1, 2
),
growth_calc AS (
    SELECT 
        *,
        LAG(current_month_spend) OVER (PARTITION BY customer_id ORDER BY sales_month) AS prev_month_spend
    FROM monthly_sales
)
SELECT 
    *,
    ((current_month_spend - prev_month_spend) / NULLIF(prev_month_spend, 0)) * 100 AS pct_growth
FROM growth_calc;

```

### Handling Edge Cases (NULLs & Skew)

* **NULLs:** Use `COALESCE(column, 0)` to prevent math errors or `IS NULL` filters.
* **Data Skew:** In joins, if one ID (like a "NULL" or "Unknown" ID) appears millions of times, it causes a "hot partition." You can handle this by filtering them out or using a "salt" (adding a random number to the join key to spread the load).

## 3. Data Modeling & System Design

### View vs. Materialized View vs. Physical Table

| Feature | View | Materialized View | Physical Table |
| --- | --- | --- | --- |
| **Storage** | None (Logical only) | Stores results on disk | Stores data on disk |
| **Performance** | Computes on the fly (Slower) | Faster (Pre-computed) | Fastest for complex reads |
| **Freshness** | Real-time | Depends on refresh policy | Depends on ETL/ELT pipeline |
| **Use Case** | Simple logic/security | Expensive aggregations | Core warehouse entities |

### Modeling Immutability vs. Corrections

To ensure quarterly revenue doesn't change historically while allowing corrections, you use **Snapshots and Audit Layers**.

1. **Immutable Snapshot:** At the end of the quarter, take a hard copy of the data (e.g., `revenue_q1_2024`). This is never touched.
2. **The "Correction" Table:** Instead of updating the snapshot, you create an `adjustments` table.
3. **Reporting View:** Create a view that `UNION ALL`s the snapshot and the adjustments. This preserves the "As-Is" history while reflecting the "To-Be" reality.

### Data Governance & PII Masking

In Snowflake, this is handled via **Dynamic Data Masking** and **Row Access Policies**.

* **Masking:** A policy is applied to a column. An Admin sees the plain text, but a regular Analyst sees `XXXX-XXXX`.
* **PII Tagging:** Use object tagging to track where sensitive data (email, SSN) lives across thousands of tables.

### Scenario: Subscription Revenue Model

**Problem:** Quarterly snapshots must remain immutable, but sales needs corrections.

**Proposed Solution:**

* **Fact Table:** A standard transaction grain table.
* **Snapshot Table:** A periodic snapshot (end-of-quarter) that is "written once."
* **Delta/Correction Table:** A table specifically for manual overrides or late-arriving dimensions, including a `created_at` timestamp and the reason for the change.
* **Semantic Layer:** A view that joins the snapshot to the delta table. If a correction exists for a specific subscription ID in the delta table, the view displays the corrected value; otherwise, it shows the snapshot value. This provides a full audit trail of *why* and *when* a number changed.
