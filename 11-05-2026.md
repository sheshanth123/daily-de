
## **I. Architecture & Design**

### **1. How would you design a real-time data pipeline for millions of financial transactions?**

**Design Goals:** Low latency, high throughput, reliability (exactly-once), auditability, compliance (PCI/GDPR), and observability.

* **Ingest:** Use a durable message bus like **Kafka (MSK)** or **Kinesis**. Partition by account/customer ID to preserve ordering.
* **Schema:** Implement **Avro/Protobuf** with a Schema Registry for strict validation.
* **Processing:** Use **Apache Flink** (for low-latency stateful ops like fraud detection) or **Spark Structured Streaming** (for micro-batching).
* **Exactly-Once Semantics:** Achieve this via checkpointing, Kafka transactions, or idempotent writes to downstream sinks.
* **Storage:**
* **OLTP:** Aurora/Postgres for real-time balances.
* **OLAP:** S3 with Iceberg/Hudi for analytical lakes.


* **Security:** TLS in transit, KMS encryption at rest, IAM roles, and field-level PII tokenization.

---

### **2. How do you ensure data consistency across distributed systems?**

* **Idempotency:** Use unique keys to ensure repeated messages don't create duplicate state.
* **CDC (Change Data Capture):** Use tools like **Debezium** to emit canonical events from source databases.
* **Distributed Transactions:** Prefer **Saga patterns** or compensating transactions over two-phase commits for better scalability.
* **Causal Consistency:** Use vector clocks or timestamps when ordering matters across services.
* **Reconciliation:** Run periodic "hash checks" or record count audits between source and target systems.

---

### **3. How would you design a monitoring system for data pipelines?**

Focus on the four pillars of observability:

* **Metrics:** Track end-to-end latency, throughput, Kafka lag, and error rates (Prometheus/Grafana).
* **Logs:** Use structured logging with **Correlation IDs** to trace events across services.
* **Tracing:** Implement **OpenTelemetry** for distributed request tracing.
* **Health Checks:** Use **synthetic transactions** to test the pipeline end-to-end regularly.
* **Alerting:** Set alerts based on **SLOs/SLAs** rather than minor transient blips.

## **II. Data Quality & Governance**

### **4. How do you handle and prevent duplicate records in production?**

* **Immediate Remediation:** Identify duplicates via composite keys; use Spark SQL `ROW_NUMBER() OVER (PARTITION BY... ORDER BY...)` to keep only the latest version.
* **Prevention at Source:** Enforce unique event IDs at the producer level.
* **Prevention at Ingest:** Use Flink state or Spark’s `dropDuplicates` with watermarks to filter duplicates within a time window.
* **Storage Layer:** Use upsert-capable formats like **Apache Iceberg** or **Hudi**.

---

### **5. How do you manage schema evolution without breaking downstream systems?**

* **Compatibility Rules:** Enforce `BACKWARD`, `FORWARD`, or `FULL` compatibility in the Schema Registry.
* **Graceful Evolution:** Add optional fields with defaults; avoid renaming or deleting fields in-place.
* **Versioning:** If a breaking change is required, publish a new version and maintain parallel topics/pipelines during a migration window.
* **Consumer Resilience:** Write "defensive" code that ignores unknown fields.

---

### **6. How do you secure PII data and ensure compliance (GDPR/CCPA)?**

* **Classification:** Automated tagging of sensitive fields.
* **Access Control:** Fine-grained **RBAC** via IAM and Lakehouse ACLs.
* **Pseudonymization:** Replace identifiers with tokens so data remains useful for analytics without exposing identities.
* **Encryption:** Use **envelope encryption** for field-level security.
* **Right to be Forgotten:** Use Iceberg/Hudi to perform efficient deletes of specific user records within the data lake.

---

## **III. Performance Tuning & Troubleshooting**

### **7. How do you debug a slow-running Spark job?**

1. **Spark UI Analysis:** Check the History Server for stage durations, **GC (Garbage Collection) time**, and memory spills.
2. **Identify Data Skew:** Look for specific tasks taking significantly longer than others.
* *Fix:* Use **salting**, repartitioning, or broadcast joins.


3. **Shuffle Optimization:** Minimize "wide" transformations. Increase `spark.sql.shuffle.partitions` if tasks are processing too much data.
4. **Resource Tuning:** Adjust executor memory and cores; switch to **Kryo serialization** to reduce overhead.

---

### **8. How do you join two huge datasets that don’t fit in memory?**

* **Sort-Merge Join:** Spark’s default for large joins; it sorts data on disk and merges it.
* **Bucketing:** Pre-bucket both tables on the join key to avoid a shuffle entirely.
* **Salting:** If one join key is skewed (e.g., a "Null" or "Default" ID), add a random suffix to distribute load.
* **Filtering:** Apply predicate pushdown to reduce data size *before* the join.

---

### **9. How do you handle late-arriving data in time-series pipelines?**

* **Watermarks:** Use event-time watermarks to define how long the system waits for late data.
* **Allowed Lateness:** Configure a window to accept data for a specific period (e.g., 1 hour) and update previous aggregates.
* **Side Outputs:** Route "extremely late" data to a Dead Letter Queue (DLQ) for manual backfilling.
* **Upserts:** Use a storage engine (Hudi/Iceberg) that allows updating existing records with late-arriving info.

---

### **10. Troubleshooting Delayed Kafka Messages**

* **Check Consumer Lag:** Identify if the lag is specific to one partition (skew) or across all (resource issue).
* **Broker Health:** Check for under-replicated partitions or CPU/Disk I/O saturation on the brokers.
* **Consumer Performance:** Profile the consumer for blocking I/O or slow downstream API calls.
* **Rebalance Storms:** Check if consumers are frequently joining/leaving the group, causing stalls.

---

## **IV. Migration & Operations**

### **11. Strategies for migrating an on-prem warehouse to the cloud**

1. **Assessment:** Inventory all ETL jobs, data sizes, and SLAs.
2. **Migration Pattern:** Choose between **Lift-and-Shift** (fast) or **Re-architecting** (Cloud-native S3/Iceberg).
3. **Data Transfer:** Use **AWS Snowball** for bulk moves and **CDC** (Change Data Capture) for incremental sync during the transition.
4. **Validation:** Run parallel systems and use checksums to ensure data parity.
5. **Cutover:** Perform a staged rollout with a clear rollback plan.

---

### **12. How do you backfill data without impacting production?**

* **Isolated Compute:** Run backfill jobs on a separate auto-scaling cluster to avoid resource contention.
* **Throttling:** Limit the concurrency of backfill tasks.
* **Atomic Swaps:** Write backfilled data to a temporary table/partition, validate it, and then use an atomic "swap" or "merge" to update production.
* **Dry Runs:** Perform a small-scale sample backfill to estimate time and resource impact.
