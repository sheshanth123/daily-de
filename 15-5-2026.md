

## 1. Uber: Toll Plaza Revenue Calculation

**Problem:** Calculate the total toll collected per day based on dynamic pricing. If the same vehicle returns within 4 hours, apply a reduced toll.

### Business Rules



* **Standard Rates:** Motorcycle (Free), Car (₹40), Bus (₹70), Truck (₹80).


* **Reduced Rates (within 4 hours):** Car (₹20), Bus (₹30), Truck (₹40).



### Engineering Insight

To solve this correctly, we must identify the previous crossing per vehicle and calculate the time difference. Using a **Window Function** (`LAG`) is ideal because it allows for a single linear scan and avoids expensive self-joins that don't scale.

### SQL Solution

```sql
WITH base AS (
    SELECT
        vehicle_no,
        vehicle_type,
        crossing_time,
        DATE(crossing_time) AS crossing_date,
        LAG(crossing_time) OVER (
            PARTITION BY vehicle_no 
            ORDER BY crossing_time
        ) AS prev_crossing_time
    FROM toll_log
),
toll_calc AS (
    SELECT *,
        CASE
            WHEN vehicle_type = 'motorcycle' THEN 0
            WHEN prev_crossing_time IS NOT NULL 
                 AND TIMESTAMPDIFF(HOUR, prev_crossing_time, crossing_time) <= 4
            THEN CASE
                    WHEN vehicle_type = 'car' THEN 20
                    WHEN vehicle_type = 'bus' THEN 30
                    WHEN vehicle_type = 'truck' THEN 40
                 END
            ELSE CASE
                    WHEN vehicle_type = 'car' THEN 40
                    WHEN vehicle_type = 'bus' THEN 70
                    WHEN vehicle_type = 'truck' THEN 80
                 END
        END AS toll_amount
    FROM base
)
SELECT
    crossing_date,
    SUM(toll_amount) AS total_toll
FROM toll_calc
GROUP BY crossing_date;

```

### PySpark Solution

```python
from pyspark.sql import functions as F
from pyspark.sql.window import Window

w = Window.partitionBy("vehicle_no").orderBy("crossing_time")

df_prev = df.withColumn("prev_time", F.lag("crossing_time").over(w))

df_diff = df_prev.withColumn(
    "hours_diff",
    (F.col("crossing_time").cast("long") - F.col("prev_time").cast("long")) / 3600
)

df_toll = df_diff.withColumn(
    "toll_amount",
    F.when(F.col("vehicle_type") == "motorcycle", 0)
     .when((F.col("hours_diff") <= 4) & F.col("prev_time").isNotNull(),
           F.when(F.col("vehicle_type") == "car", 20)
            .when(F.col("vehicle_type") == "bus", 30)
            .otherwise(40)) 
     .otherwise(
           F.when(F.col("vehicle_type") == "car", 40)
            .when(F.col("vehicle_type") == "bus", 70)
            .otherwise(80)) 
)

result = df_toll.groupBy(F.to_date("crossing_time").alias("date")).agg(F.sum("toll_amount"))
result.show()

```

## 2. Walmart: Find Palindrome Keys

**Problem:** A key is called a "palindrome key" if the sequence of values for that key reads the same forward and backward.

### Logical Approach



* **Preserve Order:** Ensure the sequence of values is maintained using an ordering column.


* **Build Sequence:** Aggregate the values into an array or sequence for each key.


* **Check Palindrome:** Verify if the sequence is equal to its reverse.



### SQL Solution

```sql
WITH ordered_data AS (
    SELECT
        key,
        value,
        ROW_NUMBER() OVER (PARTITION BY key ORDER BY key) AS rn
    FROM input_table
),
grouped_data AS (
    SELECT
        key,
        ARRAY_AGG(value ORDER BY rn) AS seq_values
    FROM ordered_data
    GROUP BY key
)
SELECT key AS palindrome_keys
FROM grouped_data
WHERE seq_values = REVERSE(seq_values);

```

### PySpark Solution

```python
from pyspark.sql import functions as F
from pyspark.sql.window import Window

window_spec = Window.partitionBy("key").orderBy("key")
df_ordered = df.withColumn("rn", F.row_number().over(window_spec))

df_grouped = df_ordered.groupBy("key").agg(
    F.collect_list("value").alias("seq_values")
)
result = df_grouped.filter(
    F.col("seq_values") == F.reverse(F.col("seq_values"))
).select("key")
result.show()

```

## 3. Amazon: Valid Triangle Detection

**Problem:** Identify combinations of side lengths (A, B, C) that can actually form a valid triangle.

### Geometry Rule



A triangle is valid if:

* $A + B > C$

* $A + C > B$

* $B + C > A$


### SQL Solution (Pivot and Validate)

```sql
SELECT
    CombinationID
FROM triangles
GROUP BY CombinationID
HAVING
    MAX(CASE WHEN SideID = 'A' THEN SideLength END) + 
    MAX(CASE WHEN SideID = 'B' THEN SideLength END) > 
    MAX(CASE WHEN SideID = 'C' THEN SideLength END)
AND
    MAX(CASE WHEN SideID = 'A' THEN SideLength END) + 
    MAX(CASE WHEN SideID = 'C' THEN SideLength END) > 
    MAX(CASE WHEN SideID = 'B' THEN SideLength END)
AND
    MAX(CASE WHEN SideID = 'B' THEN SideLength END) + 
    MAX(CASE WHEN SideID = 'C' THEN SideLength END) > 
    MAX(CASE WHEN SideID = 'A' THEN SideLength END);

```

### PySpark Solution (Pivot and Filter)

```python
from pyspark.sql import functions as F

pivot_df = (
    df.groupBy("CombinationID")
      .pivot("SideID", ["A", "B", "C"])
      .agg(F.max("SideLength"))
)

result = pivot_df.filter(
    (F.col("A") + F.col("B") > F.col("C")) &
    (F.col("A") + F.col("C") > F.col("B")) &
    (F.col("B") + F.col("C") > F.col("A"))
).select("CombinationID")

result.show()

```

## Interview Gold: Scalability & Optimization



* **Window Functions over Self-Joins:** Self-joins cause quadratic explosion and memory risks; window functions provide linear complexity and are optimizer-friendly.


* **Data Skew:** Heavy users (fleet vehicles) can cause memory pressure. Limit window scope by partitioning by both ID and Date where possible.


* **Incremental Processing:** In production, avoid recomputing full history daily. Maintain a state table of the "last crossing" per vehicle to process only new data.


* **Numeric Computation:** Convert timestamps to epoch seconds for subtraction to reduce CPU overhead in Spark compared to repeated timestamp functions.
