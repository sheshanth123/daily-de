

## Part 1: Python & Data Manipulation

### **Q1: How do you convert a dictionary into a DataFrame using Pandas?**

**Answer:**
The most direct way is using the `pd.DataFrame()` constructor. However, the method depends on the dictionary's structure (e.g., list of values vs. list of dictionaries).

```python
import pandas as pd

# 1. Dictionary of lists (Column-oriented)
data = {'Name': ['Alice', 'Bob'], 'Age': [25, 30]}
df = pd.DataFrame(data)

# 2. List of dictionaries (Row-oriented)
data_rows = [{'Name': 'Alice', 'Age': 25}, {'Name': 'Bob', 'Age': 30}]
df_rows = pd.DataFrame(data_rows)

# 3. Using from_dict for specific orientations
df_custom = pd.DataFrame.from_dict(data, orient='columns')

```

### **Q2: Explain how to slice and dice data using Pandas and NumPy.**

**Answer:**
In Pandas, the distinction between `.loc` and `.iloc` is critical for senior roles.

* **`.loc`:** Label-based indexing. Used when you know the name of the row index or column.
* **`.iloc`:** Integer-based indexing. Used when you want to select by position (e.g., "the first 5 rows").

**Sample Code:**

```python
# Slicing rows 0 to 10 and selecting specific columns
subset = df.loc[0:10, ['Name', 'Age']]

# Dicing using boolean indexing (Filtering)
filtered_df = df[(df['Age'] > 21) & (df['Dept'] == 'Engineering')]

# NumPy slicing (Start:Stop:Step)
import numpy as np
arr = np.array([1, 2, 3, 4, 5])
sliced_arr = arr[1:4] # Returns [2, 3, 4]

```

### **Q3: How do you perform a simple aggregation, like counting employees per department?**

**Answer:**
You use the `.groupby()` method combined with an aggregation function like `.count()` or `.size()`.

```python
# Simple count
dept_counts = df.groupby('Dept')['Employee_ID'].count()

# Multiple aggregations at once
dept_stats = df.groupby('Dept').agg({
    'Employee_ID': 'count',
    'Salary': ['mean', 'median']
})

```

## Part 2: SQL & Analytics

### **Q4: Calculate each customer’s monthly spend and percentage growth compared to the previous month.**

**Answer:**
This requires a **Common Table Expression (CTE)** to aggregate data first, then a **Window Function** (`LAG`) to compare periods.

```sql
WITH MonthlySpend AS (
    SELECT 
        customer_id,
        DATE_TRUNC('month', transaction_date) AS report_month,
        SUM(amount) AS total_spend
    FROM transactions
    GROUP BY 1, 2
),
GrowthCalc AS (
    SELECT 
        customer_id,
        report_month,
        total_spend,
        LAG(total_spend) OVER (PARTITION BY customer_id ORDER BY report_month) AS prev_month_spend
    FROM MonthlySpend
)
SELECT 
    customer_id,
    report_month,
    total_spend,
    ROUND(((total_spend - prev_month_spend) / NULLIF(prev_month_spend, 0)) * 100, 2) AS pct_growth
FROM GrowthCalc;

```

### **Q5: How do you handle joins across large tables with skewed data or NULL values?**

**Answer:**

* **NULLs:** Joining on columns with NULLs can lead to data loss or unexpected results. Use `COALESCE(key, 'Unknown')` or filter them out if they aren't required for the join.
* **Data Skew:** If one ID (e.g., a guest user ID '0') appears in 50% of your rows, the warehouse node processing that ID will bottleneck.
* **Solution:** "Salt" the keys by adding a random integer to the join key to redistribute the load across nodes, or handle the skewed values in a separate, smaller query.

## Part 3: Data Modeling & System Design

### **Q6: What is the difference between a View, a Materialized View, and a Physical Table?**

**Answer:**

| Feature | View | Materialized View | Physical Table |
| --- | --- | --- | --- |
| **Storage** | None (Logical only) | Result set is stored | Data is stored |
| **Performance** | Slow (Runs query every time) | Fast (Pre-computed) | Fastest for complex logic |
| **Maintenance** | None | Auto-refreshes (usually) | Manual (ETL/ELT needed) |
| **Cost** | Computing cost per run | Storage + Refresh cost | Storage + ETL compute |

### **Q7: How do you model quarterly revenue so numbers stay immutable, but corrections can be made?**

**Answer:**
Use a **Snapshot + Delta** architecture.

1. **The Snapshot:** At the end of the quarter, run an ELT job that saves the current state into a permanent table (e.g., `fact_revenue_q1_2024`). This table is never updated.
2. **The Delta Table:** Create an `adjustments` table where manual corrections are logged with a timestamp and a "reason code."
3. **The Reporting View:** Create a view that joins the two.
* `SELECT * FROM snapshot UNION ALL SELECT * FROM adjustments`
* This preserves the "Original Audit" while providing the "Corrected Reality."



### **Q8: How do you ensure Data Governance and handle PII masking?**

**Answer:**
In a modern stack like Snowflake, you utilize:

* **RBAC (Role-Based Access Control):** Ensuring only the "Finance_Admin" role can see sensitive columns.
* **Dynamic Data Masking:** Creating a policy that detects PII (like email) and masks it (e.g., `a***@email.com`) for unauthorized roles.
* **Object Tagging:** Tagging columns as `PII_Type = 'Email'` so you can track sensitive data across the entire warehouse for compliance (GDPR/CCPA).

### **Q9: Design a data model for tracking subscription revenue with immutable snapshots and occasional corrections.**

**Answer:**
For a Senior role, emphasize the **Medallion Architecture** or **Data Vault** principles:

* **Raw Layer:** Ingest subscription events (sign-ups, cancellations).
* **Silver Layer:** A Type-2 Slowly Changing Dimension (SCD) for subscriptions to track history.
* **Gold Layer (The Model):**
* A **Periodic Snapshot Fact Table** that captures the state of all subscriptions at the stroke of midnight on the last day of the quarter.
* An **Audit/Correction Table** for manual overrides.
* Final metrics are served via a **Semantic Layer** (like dbt or Looker) that logic-checks the snapshot against corrections to ensure the "Official" quarterly report matches the ledger.
