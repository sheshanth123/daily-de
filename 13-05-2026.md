

## 1. The "Why Spark?" Question

**Question:** "Why did you choose Spark for this specific project instead of a standard SQL engine or Pandas?"

**The Answer:**
Focus on the **memory bottleneck** and **scalability**. Pandas is single-node and loads data into RAM; Spark handles distributed memory.

* **Green Flag:** "We hit a wall at 50GB. Pandas would OOM (Out of Memory). We chose Spark to distribute the workload across a cluster, accepting the overhead of network shuffles for the ability to process multi-terabyte datasets."

**Code Comparison (Pandas vs. Spark):**

```python
# Pandas - Fails if data > RAM
import pandas as pd
df = pd.read_csv("massive_data.csv") 

# Spark - Scalable & Distributed
from pyspark.sql import SparkSession
spark = SparkSession.builder.appName("ScalableJob").getOrCreate()
df = spark.read.parquet("s3://bucket/massive_data/")

```

## 2. Pipeline Design & Modularity

**Question:** "How do you structure your Spark jobs? Tell me about your pipeline stages."

**The Answer:**
Talk about **separation of concerns** and **checkpoints**.

* **Green Flag:** "I don't write one giant script. I build composable transformations. Each stage (Clean, Dedupe, Enrich) is isolated so that a failure in the final enrichment stage doesn't require re-running the heavy ingestion logic."

**Code Snippet (Composable Logic):**

```python
def clean_data(df):
    return df.dropDuplicates().fillna(0)

def enrich_data(df, mapping_df):
    return df.join(mapping_df, "id", "left")

# Pipeline Flow
raw_df = spark.read.table("raw_source")
cleaned_df = clean_data(raw_df)
# We could save cleaned_df to a temporary path here as a checkpoint
final_df = enrich_data(cleaned_df, lookup_df)

```

## 3. Handling Data Skew & Failures

**Question:** "Describe a time a Spark job failed and how you resolved it."

**The Answer:**
Focus on **Data Skew**. Skew happens when one partition is much larger than others, causing one task to hang or OOM while others finish.

* **Green Flag:** "We had a join that always failed. I checked the Spark UI and saw one executor was processing 10x more data than others due to a null key. I fixed this by 'salting' the join key to redistribute the data."

**Code Snippet (Salting for Skew):**

```python
from pyspark.sql import functions as F

# Adding a random 'salt' to the key to break up the large partition
skewed_df = skewed_df.withColumn("salt", F.rand() * 10)
skewed_df = skewed_df.withColumn("salted_key", F.concat(F.col("id"), F.col("salt")))

# The lookup table must be exploded to match the salted keys
lookup_df = lookup_df.withColumn("salt", F.explode(F.array([F.lit(i) for i in range(10)])))
lookup_df = lookup_df.withColumn("salted_key", F.concat(F.col("id"), F.col("salt")))

joined_df = skewed_df.join(lookup_df, "salted_key")

```

## 4. Idempotency & Reruns

**Question:** "What is your strategy for re-running a failed job that partially wrote data?"

**The Answer:**
The keyword is **Idempotency**. Your job should be able to run multiple times and produce the same result without duplicating data.

* **Green Flag:** "I use partition-level overwrites. If the job fails halfway, a re-run will simply overwrite the specific partition (e.g., `year=2024/month=01`) instead of appending duplicates."

**Code Snippet (Atomic Overwrites):**

```python
# Instead of .mode("append"), use "overwrite" with partition discovery
df.write \
    .mode("overwrite") \
    .partitionBy("process_date") \
    .option("partitionOverwriteMode", "dynamic") \
    .save("s3://my-warehouse/table/")

```

## 5. Shuffle Optimization (Broadcast Joins)

**Question:** "When is a shuffle expensive, and how can you avoid it?"

**The Answer:**
Shuffling is the process of moving data across the network between executors. It is the biggest performance killer in Spark.

* **Green Flag:** "Standard joins cause a shuffle. If one table is small (e.g., a few hundred MBs), I use a Broadcast Join to send that small table to every executor, eliminating the need to move the large table across the network."

**Code Snippet (Broadcast Join):**

```python
from pyspark.sql.functions import broadcast

# Prevents the large 'sales_df' from being shuffled
# only the small 'store_metadata' is sent to all nodes
optimized_df = sales_df.join(broadcast(store_metadata), "store_id")

```

## 6. Storage & Physical Layout

**Question:** "How does the way you store data affect your Spark job's performance?"

**The Answer:**
Talk about **Predicate Pushdown** and **File Formats**.

* **Green Flag:** "We use Parquet because it’s columnar. By partitioning data by `date` and sorting it by `user_id` (Z-Ordering), Spark can skip reading 90% of the files that don't match the query filter."

**Code Snippet (Partitioning & Sorting):**

```python
# Optimizing for downstream 'user_id' lookups
df.write \
    .partitionBy("region") \
    .sortBy("user_id") \
    .format("parquet") \
    .save("s3://data/refined_users")

```

## 7. Operational Maturity & Maintainability

**Question:** "What do you do to make your Spark code 'production-ready'?"

**The Answer:**
Focus on **Schema Validation** and **Monitoring**.

* **Green Flag:** "I implement 'Fail-Fast' logic. I validate the schema at the ingestion point using a defined StructType. If the upstream source changes a column from Integer to String, the job stops immediately rather than producing corrupt data downstream."

**Code Snippet (Strict Schema Validation):**

```python
from pyspark.sql.types import StructType, StructField, StringType, IntegerType

# Explicitly defining the expected schema
expected_schema = StructType([
    StructField("user_id", IntegerType(), False),
    StructField("email", StringType(), True)
])

# Loading with the schema ensures immediate failure or nulling of bad rows
df = spark.read.schema(expected_schema).json("s3://raw/data.json")

```
